Vom Setup zum fertigen Plan

Ein achtstufiges Vorgehen, um mit Claude Code eine schlanke KI-App zur Projektplanung aufzubauen — von der Projekteinrichtung bis zum API-Schlüssel.

Blättern Sie sich Schritt für Schritt durch das Vorgehen. Wo Texte zweisprachig vorliegen, stehen die englische und die deutsche Fassung nebeneinander zum direkten Vergleich. Code-Befehle lassen sich per Klick kopieren.

Tipp: Navigieren Sie mit den Pfeiltasten ← → oder den Buttons unten.

Einrichten eines Next.js-Projekts

Erstelle ein Next.js-Projekt. Nutze den npx-Befehl dafür.

Schritt 2 von 8
02

CLAUDE.md / AGENTS.md

Lege die zentralen Regeln für den Coding-Agenten fest. Englische und deutsche Fassung im Vergleich:

English

# CRITICAL RULES — MUST FOLLOW

## RESPONSES

  • Keep responses concise and to the point — unless the user asks otherwise

## PLANNING MODE

  • Always ask clarifying questions
  • Never assume design, tech stack or features
  • Use deep-dive sub-agents to assist with research
  • Use deep-dive sub-agents to review the different aspects of your plan before presenting to the user

## CHANGE / EDIT MODE

  • Never implement features yourself when possible — use sub-agents!
  • Identify changes from the plan that can be implemented in parallel, and use sub-agents to implement the features efficiently
  • When using sub-agents to implement features, act as a coordinator only
  • Use the best model for the task — premium models for complex tasks (like coding) and mid-tier models for simpler tasks, like documentation
  • After completing features (large or small), always run commands like lint, type check and next build to check code quality

## DATABASE SCHEMA CHANGES

  • Whenever you make changes to the database schema, ALWAYS run the drizzle generate and migrate commands
  • NEVER run drizzle push!

## TESTING

  • Use any testing tools, libraries available to the project for testing your changes
  • Never assume your changes simply work, always test!
  • If the project does not have any testing tools, scripts, MCP tools, skills, etc. available for testing, ask the user whether testing should be skipped.

## UI DESIGN

  • Always follow the UI design system when creating or reviewing components or pages.
  • Design System: @DESIGN.md
Deutsch

# WICHTIGE REGELN – UNBEDINGT EINHALTEN

## ANTWORTEN

  • Halte die Antworten prägnant und auf den Punkt gebracht – es sei denn, der Benutzer wünscht etwas anderes

## PLANUNGSMODUS

  • Stelle immer klärende Fragen
  • Gehe niemals von bestimmten Designs, Tech-Stacks oder Funktionen aus
  • Setze Deep-Dive-Subagenten ein, um die Recherche zu unterstützen
  • Setze Deep-Dive-Subagenten ein, um die verschiedenen Aspekte deines Plans zu überprüfen, bevor du ihn dem Benutzer präsentierst

## ÄNDERUNGS-/BEARBEITUNGSMODUS

  • Implementiere Funktionen nach Möglichkeit niemals selbst – setze Subagenten ein!
  • Identifiziere Änderungen am Plan, die parallel umgesetzt werden können, und setze Subagenten ein, um die Funktionen effizient zu implementieren
  • Wenn du Subagenten zur Implementierung von Funktionen einsetzt, fungiere ausschliesslich als Koordinator
  • Verwende das für die Aufgabe am besten geeignete Modell – Premium-Modelle für komplexe Aufgaben (wie das Programmieren) und Modelle der mittleren Leistungsklasse für einfachere Aufgaben, wie die Dokumentation
  • Führe nach Fertigstellung von Funktionen (ob gross oder klein) immer Befehle wie «lint», «type check» und «next build» aus, um die Codequalität zu überprüfen

## ÄNDERUNGEN AM DATENBANKSCHEMA

  • Führe bei jeder Änderung am Datenbank-Schema IMMER die Befehle «drizzle generate» und «migrate» aus
  • Führen Sie NIEMALS «drizzle push» aus!

## TESTEN

  • Nutze alle im Projekt verfügbaren Testtools und Bibliotheken, um deine Änderungen zu testen
  • Gehe niemals davon aus, dass deine Änderungen einfach so funktionieren – teste immer!
  • Falls im Projekt keine Testtools, Skripte, MCP-Tools, Kenntnisse usw. zum Testen verfügbar sind, frage den Benutzer, ob das Testen übersprungen werden soll.

## UI-DESIGN

  • Halte dich beim Erstellen oder Überprüfen von Komponenten oder Seiten stets an das UI-Designsystem.
  • Design System: @DESIGN.md

06

PROMPT.md — der Projektauftrag

Beschreibe die zu bauende App. Englische und deutsche Fassung im Vergleich:

English

We're building a lightweight AI Project Planner app for this tutorial. The app should let a user enter a rough app idea, then generate an editable project brief with:

  • app summary
  • target users
  • core features
  • recommended tech stack
  • pages/routes
  • possible data model
  • build phases
  • risks/edge cases
  • final copyable starter prompt for a coding agent

Keep the scope small: no auth, no database, no payments. Use Next.js, Tailwind, and shadcn/ui. For now, plan the architecture, components, routes, data flow, and implementation steps. We will be using the AI SDK with OpenRouter for inference.

For the data model, it would be cool if you could visualize the relationship between the different entities as well (perhaps reactflow or something).

Deutsch

Für dieses Tutorial entwickeln wir eine schlanke KI-App zur Projektplanung. Die App soll es dem Benutzer ermöglichen, eine grobe App-Idee einzugeben, und anschliessend ein bearbeitbares Projektdossier erstellen, das Folgendes enthält:

  • Zusammenfassung der App
  • Zielgruppe
  • Kernfunktionen
  • Empfohlener Tech-Stack
  • Seiten/Routen
  • Mögliches Datenmodell
  • Entwicklungsphasen
  • Risiken/Grenzfälle
  • Abschliessende, kopierbare Startanweisung für einen Programmieragenten

Halte den Umfang klein: keine Authentifizierung, keine Datenbank, keine Zahlungen. Verwende Next.js, Tailwind und shadcn/ui. Plane vorerst die Architektur, die Komponenten, die Routen, den Datenfluss und die Implementierungsschritte. Wir werden das AI-SDK mit OpenRouter für die Inferenz nutzen.

Was das Datenmodell betrifft, wäre es cool, wenn du auch die Beziehungen zwischen den verschiedenen Entitäten visualisieren könntest (vielleicht mit ReactFlow oder etwas Ähnlichem).
